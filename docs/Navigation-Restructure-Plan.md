# Picko 导航与整理工作台重构计划

更新日期：2026-06-21

## 1. 背景

当前 iOS 底部 Tab 为：

```text
首页 / 复核 / 相似 / 预删除篮
```

这个结构的问题是底部 Tab 混合了三种不同层级：

1. `首页` 是入口聚合。
2. `复核` 和 `相似` 是两种整理方式。
3. `预删除篮` 是整理后的结果区和安全确认区。

同时，`时间` 和 `地点` 也是整理方式，但它们被放在首页二级入口里，没有和 `复核`、`相似` 处在同一层级。用户会难以理解这些入口之间的关系：哪些是流程阶段，哪些是整理方式，哪些是结果状态。

## 2. 已确认方向

第一版导航重构采用：

```text
建议 / 整理 / 预删除篮 / 设置
```

职责定义：

1. `建议`：Picko 主动告诉用户现在值得整理什么，承载继续整理、近期发现、相似组摘要、时间地点摘要等。
2. `整理`：所有整理方式的统一工作台，包含单张复核、相似照片、按时间、按地点。
3. `预删除篮`：恢复、空间预估、最终交由系统 Photos 确认。
4. `设置`：手势方向、权限状态、隐私说明、本地整理状态清理等全局偏好。

核心原则：

- 底部 Tab 表达用户阶段，不表达具体算法或整理方式。
- `复核` 不再作为底部 Tab，而是 `整理` 内部的一种整理方式。
- `相似 / 时间 / 地点 / 单张复核` 在 `整理` 内部同层。
- `预删除篮` 保持独立，因为它是删除前安全边界。

## 3. 整理 Tab 第一版结构：A + C Lite

整理 Tab 第一版采用 `A + C Lite` 混合版：

```text
整理
├─ 继续整理 / 推荐开始
├─ 单张复核
├─ 相似照片
├─ 按时间
└─ 按地点
```

### 3.1 为什么不是纯 A

纯 A 是四个入口卡片：

```text
整理
├─ 单张复核
├─ 相似照片
├─ 按时间
└─ 按地点
```

它的优点是清楚、稳定、迁移成本低，但 Picko 的主动帮助感较弱，像一个工具箱。

### 3.2 为什么不是纯 C

纯 C 是主动推荐队列优先：

```text
整理
├─ 推荐队列
├─ 相似组
├─ 按时间
└─ 按地点
```

它的优点是更聪明、更像 Picko 在帮用户安排整理顺序，但第一版如果没有足够透明的推荐规则，用户可能不理解为什么 Picko 让他先处理某一批照片。

### 3.3 A + C Lite 的第一版目标

第一版保持 A 的清晰结构，同时在顶部增加轻量推荐入口：

```text
整理
├─ 顶部主卡：继续整理 / 推荐开始
│  ├─ 有未完成 scope → 继续整理
│  ├─ 有高置信相似组 → 建议先整理相似照片
│  ├─ 有近期未复核照片 → 建议开始单张复核
│  └─ 无明确推荐 → 选择一种方式开始
├─ 整理方式卡片
│  ├─ 单张复核
│  ├─ 相似照片
│  ├─ 按时间
│  └─ 按地点
└─ 底部安全提示：所有删除会先进入预删除篮
```

这个结构的目标是：

1. 用户可以明确知道所有整理方式在哪里。
2. Picko 仍然能给一个“先从这里开始”的轻建议。
3. 后续可以平滑升级为完整主动推荐队列。

## 4. 未来目标：演进到 C 主动推荐

需要在计划中保留的长期目标：

> Picko 最终应从“整理方式入口集合”演进为“主动推荐整理队列”，让用户进入整理页后先看到最值得处理的任务，再按需切换到具体整理方式。

### 4.1 主动推荐队列的形态

未来 C 版本的目标结构：

```text
整理
├─ 推荐队列
│  ├─ 继续整理上次未完成合集
│  ├─ 高置信相似组
│  ├─ 近期新增照片
│  ├─ 明确时间合集
│  ├─ 明确地点合集
│  └─ 截图 / 大文件 / 视频等后续整理类型
├─ 所有整理方式
│  ├─ 单张复核
│  ├─ 相似照片
│  ├─ 按时间
│  └─ 按地点
└─ 预删除篮状态摘要
```

### 4.2 推荐规则必须透明

推荐队列第一版不使用黑盒 AI。推荐原因必须能解释给用户，例如：

1. `继续上次整理`：存在未完成的 review scope 或 session。
2. `相似照片较多`：存在未复核相似组，且组内照片数达到阈值。
3. `最近新增`：近期照片未复核，用户记忆成本较低。
4. `地点清晰`：有可命名地点合集，适合按旅行或地点整理。
5. `时间集中`：某一天或某个月未复核照片数量明显集中。
6. `可节省空间较高`：候选项估算体积较大，但文案仍强调“复核后确认”，不直接催促删除。

每条推荐必须展示可理解的 reason，例如：

```text
建议先整理相似照片
发现 6 组相似照片，保留好照片后其余会先进入预删除篮。
```

### 4.3 主动推荐不能破坏用户控制感

未来 C 版本仍必须保留所有整理方式入口。推荐队列只是默认建议，不应隐藏手动选择入口。

禁止的设计：

1. 只展示推荐队列，不展示整理方式。
2. 用不可解释的“智能推荐”替代明确原因。
3. 把“节省空间”作为唯一优先级，导致产品变成 aggressive cleaner。
4. 未经用户进入预删除篮最终确认就触发系统 Photos 删除。

## 5. 分阶段实施计划

### Phase 1: 导航结构落地

目标：

1. 将底部 Tab 调整为 `建议 / 整理 / 预删除篮 / 设置`。
2. 新增 `OrganizeView` 作为整理工作台。
3. 新增 `SettingsView` 作为全局设置页。
4. 将当前复核页从 Tab 移入整理工作台的二级流程。
5. 保留预删除篮独立 Tab。

验收：

1. 底部 Tab 不再出现 `复核` 和 `相似`。
2. `整理` 页能进入单张复核、相似照片、时间、地点。
3. 现有复核、相似、时间、地点行为不被破坏。
4. 手势设置入口从复核页右上角迁移到设置页，复核页只保留和当前复核动作强相关的控件。

### Phase 2: 建议页重塑

目标：

1. 将当前首页重命名为 `建议`。
2. 首页顶部仍可保留品牌标题 `拾影`。
3. 页面内容从“入口堆叠”改为“Picko 建议你现在处理什么”。
4. 时间、地点、相似、复核入口摘要都可以出现，但点击后统一进入 `整理` 对应方式。

验收：

1. `建议` 页不再像功能菜单，而像任务摘要。
2. 与 `整理` 页职责不重复：建议负责推荐起点，整理负责方式选择。
3. 预删除篮状态可在建议页提示，但最终操作仍进入预删除篮。

### Phase 3: A + C Lite 推荐开始

目标：

1. 在 `整理` 顶部增加 `继续整理 / 推荐开始` 主卡。
2. 推荐逻辑先使用确定性优先级，不引入复杂 ranking 引擎。
3. 推荐文案必须说明原因。

初始优先级：

```text
未完成 scope/session
> 高置信相似组
> 近期未复核照片
> 时间集中合集
> 地点清晰合集
> 默认单张复核
```

验收：

1. 当存在可继续整理内容时，主卡显示“继续整理”。
2. 当存在相似组时，主卡可显示“建议先整理相似照片”。
3. 当没有明确推荐时，主卡显示“选择一种方式开始”。
4. 主卡点击后的行为可预测，不改变用户已选择的整理方式。

### Phase 4: 完整 C 主动推荐队列

目标：

1. 新增 `RecommendationQueue` 或等价 presentation model。
2. 将推荐项抽象为稳定类型，例如：

```text
ReviewRecommendation
├─ id
├─ title
├─ reason
├─ destination
├─ priority
├─ assetCount
└─ estimatedBytes?
```

3. 支持多个推荐项排序展示。
4. 保留“所有整理方式”作为 fallback。

验收：

1. 推荐队列展示多个可解释任务。
2. 每个推荐项都能进入明确的整理范围或整理方式。
3. 推荐排序有单元测试覆盖。
4. 用户可以忽略推荐，手动选择单张、相似、时间、地点。

## 6. 测试与验证要求

每个实施阶段至少覆盖：

1. `swift test`
2. iOS UI tests 覆盖底部 Tab 名称和关键入口。
3. `scripts/audit-privacy-logging.sh`
4. 模拟器截图验证：
   - 建议页
   - 整理页
   - 设置页
   - 从整理页进入单张复核
   - 从整理页进入相似照片
   - 从整理页进入时间 / 地点
5. iPhone 小屏幕检查，特别是 Tab 文案、整理卡片和空状态。

## 7. 暂不纳入本轮

1. 不引入云端 AI 或远程推荐服务。
2. 不做复杂图片质量评分 UI。
3. 不改变 Photos 删除安全边界。
4. 不把 macOS 侧强行同步为同一套底部 Tab；macOS 仍可保持 workbench/Sidebar 模式，但应同步概念命名。
